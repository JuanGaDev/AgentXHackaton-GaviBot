TEST 1 — Simple / P3 (Frontend, low impact)
------------------------------------------------------------------------
TITLE:
Product images not loading on mobile browsers

DESCRIPTION:
Several users reported that product images are not displaying on mobile
devices when using Safari on iOS 17. The images show a broken icon
placeholder instead of the actual product photo. Desktop browsers work
fine. The issue started appearing today after the latest deployment.
No console errors have been reported but the images seem to return a
404 when accessed from mobile user agents.

TEST 2 — Moderate / P2 (Backend, order processing)
------------------------------------------------------------------------
TITLE:
Order confirmation emails not being sent after checkout

DESCRIPTION:
Since this morning, customers are completing checkout successfully and
seeing the confirmation page, but they are not receiving the order
confirmation email. The orders are being created correctly in the
database and payments are being captured. Our support team has received
about 30 complaints in the last 2 hours. The issue seems to affect all
order types. We checked the background job queue and noticed the
ActionMailer jobs are piling up with a "SMTP connection timeout" error.
The SMTP provider status page shows no incidents on their end.

EXPECTED RESULT: P2, team: backend
------------------------------------------------------------------------

TEST 5 — Complex / P1 (Multi-system, ambiguous root cause)
------------------------------------------------------------------------
TITLE:
Intermittent checkout failures and inventory discrepancies during flash sale

DESCRIPTION:
During today's flash sale event (started 11:00 UTC, 15,000 concurrent
users), we are experiencing two interrelated issues that may share the
same root cause:

1. CHECKOUT FAILURES (~12% error rate):
   POST /api/v2/storefront/orders/:number/complete returns 422 with
   "Spree::Order::InsufficientStock" for items that appear in stock in
   the UI. The error is intermittent — the same item succeeds on retry
   60% of the time.

2. INVENTORY DISCREPANCIES:
   Spree::StockItem counts in the DB are going negative for high-demand
   variants. Example: variant_id=8821 showing stock_count=-14.
   The on_hand calculation in Spree::Stock::Quantifier is returning
   stale values despite cache invalidation calls.

Relevant logs:
  WARN: Optimistic locking conflict on Spree::StockItem id=4421
  lock_version expected 23 got 19 — retrying (attempt 3/3)
  ERROR: Retry limit exceeded for StockItem update, raising to caller

  Sidekiq job StockSyncWorker showing 847 retries in dead queue.
  Redis latency spiked to 340ms avg (normal: 4ms).

We suspect a race condition in the inventory reservation system under
high concurrency, possibly combined with a Redis performance issue
causing cache stampede on the stock count keys. The StockSyncWorker
jobs failing silently may have made the problem worse over time.
There is also a new "reserve stock on cart add" feature that shipped
Monday — it may be interacting badly with the checkout stock check.

EXPECTED RESULT: P1, team: backend (or database), complex multi-cause
